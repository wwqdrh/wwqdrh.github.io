# 分布式锁

redis中实现分布式锁一般依靠的是SETNX命令

> 核心在于redis是单线程的，所有的命令都是串行执行，所以在并发执行setnx的时候也只有一个会执行成功并返回1

SETNX 命令的作用是：如果指定的 key 不存在，则创建并为其设置值，然后返回状态码 1；如果指定的 key 存在，则直接返回 0。如果返回值为 1，代表获得该锁；此时其他进程再次尝试创建时，由于 key 已经存在，则都会返回 0 ，代表锁已经被占用。

## 单机环境

然后获取到锁的程序执行完了之后再通过`del`命令将key删除，这样其他工作进程就可以继续抢占

但是这里存在的风险就是如果获取到锁的挂掉了无法去执行del命令，那么就会造成死锁。解决方法是为锁设置超时时间

`EXPIRE key seconds`

> setnx + expire 操作是非原子性的，如果在设置超时时间前，程序被异常终止，则程序就会出现死锁。此时可以将 SETNX 和 EXPIRE 两个命令写在同一个 Lua 脚本中, 由redis来保证整个lua脚本的原子性

使用超时机制会存在的问题

1、事务a还没执行完，但是锁过期了被其他工作进程获取到了，就会有竞态问题
2、事务a这时候执行完了，但是锁已经是其他工作进程创建的，事务a给删除了，锁又失效了

解决方法是获取到锁的工作进程不断去续租，假设超时30秒，那么就可以每十秒去重置过期时间。问题2则是设置一个唯一值，要删除锁的时候先判断值是否相等，相等才是当前工作进程创建的，就可以删除

## 集群环境

> 哨兵模式是主从模式的升级版，能够在故障发生时自动进行故障切换，选举出新的主节点。

由于 Redis 的复制机制是异步的，因此在哨兵模式下实现的分布式锁是不可靠的

1、由于主从之间的复制操作是异步的，当主节点上创建好锁后，此时从节点上的锁可能尚未创建。而如果此时主节点发生了宕机，从节点上将不会创建该分布式锁；
 2、从节点晋升为主节点后，其他进程（或线程）仍然可以在该新主节点创建分布式锁，此时就存在多个进程（或线程）同时进入了临界区，分布式锁就失效了。

> 核心就是哨兵模式数据复制时异步的，非原子的，也就是要实现高可用的分布式锁可以使用另外的方案-集群模式


### RedLock

核心在于N个节点，客户端需要在至少`N/2+1`个实例上成功创建了锁才算获取了锁

可能客户端 A 在一半节点上创建了锁，而客户端 B 在另外一半节点上创建了锁，那么两个客户端都将无法获取到锁。如果并发很高，则可能存在多个客户端分别在部分节点上创建了锁，而没有一个客户端的数量超过 N/2+1。这也就是上面过程的最后一步中，强调一旦客户端失败后，需要等待一个随机时间后再进行重试的原因，如果是一个固定时间，则所有失败的客户端又同时发起重试，情况就还是一样

另外实现 RedLock 方案的客户端与所有 Redis 实例进行通讯时，必须要保证低延迟，而且最好能使用多路复用技术来保证一次性将 SET 命令发送到所有 Redis 节点上，并获取到对应的执行结果

*持久化*

为了保证高可用，所有 Redis 节点还需要开启持久化。假设不开启持久化，假设进程 A 获得锁后正在处理业务逻辑，此时节点宕机重启，因为锁数据丢失了，其他进程便可以再次创建该锁，因此所有 Redis 节点都需要开启 AOF 的持久化方式。

AOF 默认的同步机制为 `everysec`，即每秒进程一次持久化，此时能够兼顾性能与数据安全，发生意外宕机的时，最多会丢失一秒的数据。但如果碰巧就是在这一秒的时间内进程 A 创建了锁，并由于宕机而导致数据丢失。此时其他进程还可以创建该锁，锁的互斥性也就失效了。想要解决这个问题有两种方式：

-   **方式一**：修改 Redis.conf 中 `appendfsync` 的值为 `always`，即每次命令后都进行持久化，此时会降低 Redis 性能，进而也会降低分布式锁的性能，但锁的互斥性得到了绝对的保证；
-   **方式二**：一旦节点宕机了，需要等到锁的超时时间过了之后才进行重启，此时相当于原有锁自然失效（但你首先需要保证业务能在设定的超时时间内完成），这种方案也称为延时重启。
