使用消息队列需要注意的就是: 消息丢失/消息重复/消费顺序/消息堆积

## 讲讲rabbitmq

简单队列、发布订阅、主题、路由。核心在于使用的交换机不同，有直连，有扇形、主题、header头

routing模式下，如果有多个消费者绑定同一个queue那么能被消费几次呢，那肯定还是一次，本质上看能消费几次就是有几个队列

那如果想多消费几次就只有绑定多个queue，为每个queue绑定一个名字，这样消息复制着进入这些队列，在消费端看就相当于消费了n次

*保证消息的可靠传递或回退*

发送端收到MQ节点(Broker)的确认应答以及完善消息的补偿机制(入库，标记消息是否发送完成或者进行延迟消息确认)

- 事务机制: txSelect 将当前channel设置为transaction模式, txCommit提交事务，txRollback事务回滚
- confirm模式: 生产者投递消息后，如果Broker收到消息, 则会给我们产生一个应答用来确认消息是否发送到了Broker
	- 在channel上开启确认模式 : channel.confirmSelect()
	- 在channel上添加监听 : addConfirmListener, 监听成功和失败的返回结果, 根据具体的结果对消息进行重新发送, 或记录日志等后续处理
- return机制: 处理一些不可路由的消息，当前的exchange不存在或者指定的路由key路由不到就要使用Return Listener。Mandatory 如果为true, 则监听器会接收到路由不可达的消息, 然后进行后续处理(补偿或人工处理), 如果为false, 那么broker端自动删除该消息。

*解决消息堆积问题*

一种是速率不一致导致的问题，提供消费速度或者降低发送速度

是否使用消息超时丢弃机制主要考虑业务中是否有其他机制(例如数据入库)保证能够知道数据是否消费过了

另一种是下游服务异常退出，需要依靠为队列设置最大长度来限制

> 由客户端申明queue的时候指定或者服务端指定，两者取最小值，未确认的消息不计入限制

```bash
rabbitmqctl set_policy my-pol "^two-messages$" '{"max-length": 2, "overflow":"reject-publish"}' --apply-to queues

// 或者在使用queue的时候添加可选参数
```

默认超出长度是将消息丢弃，可以修改其行为，例如`overflow` 设置为 `reject-publish`，则最近发布的将丢弃，并在confirm机制启用了的情况下，通过 `basic.nack` 消息对发布者进行拒绝通知
