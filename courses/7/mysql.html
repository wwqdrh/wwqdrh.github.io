explain: 模拟Mysql优化器是如何执行SQL查询语句的

mysql> explain select * from staff;
+----+-------------+-------+------+---------------+------+---------+------+------+-------+
| id | select_type | table | type | possible_keys | key  | key_len | ref  | rows | Extra |
+----+-------------+-------+------+---------------+------+---------+------+------+-------+
|  1 | SIMPLE      | staff | ALL  | NULL          | NULL | NULL    | NULL |    2 | NULL  |
+----+-------------+-------+------+---------------+------+---------+------+------+-------+
1 row in set
  • id的值越大优先级越高，越先被执行
  • select_type表示查询类型，例如执行子查询或union等查询类型
  • type: 显示查询使用了何种类型, all表示全表扫描性能最差

system，const，eq_ref，ref，fulltext，ref_or_null，index_merge，unique_subquery，index_subquery，range，index，ALL

  • possible_keys: 查询可能使用到的索引都会在这里列出来
  • key: 查询真正使用到的索引。
  • rows: rows 也是一个重要的字段。 这是mysql估算的需要扫描的行数
  • Extra
    • distinct：在select部分使用了distinc关键字
    • Using filesort：当 Extra 中有 Using filesort 时, 表示 MySQL 需额外的排序操作, 不能通过索引顺序达到排序效果. 一般有 Using filesort, 都建议优化去掉, 因为这样的查询 CPU 资源消耗大.
    • Using index: "覆盖索引扫描", 表示查询在索引树中就可查找所需数据, 不用扫描表数据文件, 往往说明性能不错
    • Using temporary: 查询有使用临时表, 一般出现于排序, 分组和多表 join 的情况, 查询效率不高, 建议优化.

2、mysql索引结构，B+树细节优势，和哈希索引的区别，是为了解决什么问题？

btree索引： b+数 hash索引: hash桶 全文索引: b+锁

  • 哈希表这种结构适用于只有等值查询的场景，比如Memcached及其他一些NoSQL引擎。
  • 有序数组索引使用二分复杂度在O(logn), 但是只适用于静态存储引擎，不适合修改
  • 自平衡二叉搜索树，O(logn), 插入数据时要让二叉搜索树自平衡才能保证查询效率，更新的时间复杂度也是O(logn), 但是树的高度太高，树高20。一次查询可能需要访问20个数据块。

为了让一个查询尽量少地读磁盘，就必须让查询过程访问尽量少的数据块。那么，我们就不应该使用二叉树，而是要使用“N叉”树。这里，“N叉”树中的“N”取决于数据块的大小。以InnoDB的一个整数字段索引为例，这个N差不多是1200。这棵树高是4的时候，就可以存1200的3次方个值，这已经17亿了。考虑到树根的数据块总是在内存中的，一个10亿行的表上一个整数字段的索引，查找一个值最多只需要访问3次磁盘。

7、什么是联合索引，联合索引什么情况下会失效

在 MySQL 中，联合索引指的是将多个列组合起来创建的索引。与单列索引不同，联合索引可以提高多列组合查询的性能。

联合索引的效率取决于多个列的组合方式。当查询语句中包含联合索引中的所有列，并按照相同的顺序查询时，联合索引会非常有效。但是，如果查询语句中只包含联合索引中的一部分列或者以不同的顺序查询，联合索引可能会失效，这时需要考虑单列索引或调整查询语句的顺序。

此外，如果一个表中的数据分布不均匀，那么联合索引也可能会失效。例如，如果一个表中的 A 列有 100 个不同的值，但 B 列只有 2 个不同的值，那么以 A 列为条件进行查询时，联合索引会非常有效；但以 B 列为条件进行查询时，联合索引会失效。在这种情况下，可以考虑将 A 列和 B 列分别创建单列索引，或者将 A 列和 B 列的顺序调换。

4、什么是快照读，什么是当前读

如果需要读取历史数据或保证一致性读取，应该使用快照读；如果需要修改数据或需要提高读取速度，应该使用当前读。

在数据库中，快照读指的是可以读数据的某个时间点的操作。在快照读期间，即使其他会话已经更新了数据，快照读也会返回未更改的数据。这是因为快照读会在内部创建一个数据快照，以便在读取数据时查询快照而不是实际数据。

快照读是一种非阻塞的读取操作，可以在读取数据的同时允许其他会话继续更新数据。在 MySQL 中，有两种快照读方式：SELECT ... FROM ... WHERE ... 查询和快照读事务。

通过SELECT ... FROM ... WHERE ...查询执行的快照读会读取数据表的一个静态副本，因此不需要在读取期间对表进行锁定。但是，由于该查询不能保证读取到的数据是最新的，因此它可以被看作是读取提交数据的一种方式。

快照读事务则是在指定的时间点创建一个事务快照，并对快照进行读取操作。在该快照中，快照读事务可以读取数据库在指定时间点之前提交的数据，而无需锁定表。该事务不会受当前事务的影响，并保证在事务开始时读取到事务开始时的数据状态。该快照读事务可以在指定时间点之前写入的所有事务完成后获取一个快，因此它是一种可重复读操作。

总的来说，快照读是一种不锁定数据库表的读取操作，它会返回一个在读取时的静态副本，通常被用于查询历史数据或长时间运行的报表查询，以及保证一致性读取。

在数据库中，当前读是指在读取数据期间锁定被查询的行或表的读取方式。在当前读期间，正在读取的行或表将不能被其他会话并发地修改或删除。

当前读可以通过以下两种方式进行：

共享锁（Shared Locks）：也称读锁，多个会话可以同时获得共享锁，但任何一个会话的修改和删除操作都将被阻塞，直到共享锁被全部清除。

排他锁（Exclusive Locks）：也称写锁，只有一个会话可以获得排他锁，且在持有写锁期间，其他会话的读写操作都会被阻塞。

当前读是数据操作的一种保护机制，它可以确保被查询的数据在读取期间不会被其他并发操作修改或删除。但是，也需要注意当前读可能会导致其他会话的阻塞，在高并发的情况下可能会影响数据库的性能。

在 MySQL 中，默认情况下，SELECT语句使用的是共享锁，这称为共享读（Shared Read）。如果需要在当前读期间修改数据，可以使用FOR UPDATE或FOR SHARE选项强制使用排他锁或共享锁。但是，在进行当前读操作时，应该注意锁定被查询的数据行或表的时间，以免影响其他会话的操作。

事务

1、事务四个特性四个隔离级别

四个特性: A(原子性) C(一致性) I(隔离性) D(持久性)

其中一致性是依靠另外三个确定的

原子性: undo log, 发现错误可以进行回滚 隔离性: 锁，避免不同事务相互影响 持久性: redo log, 先写日志，并且与bin log二阶段提交

2、事务的隔离级别

  • ru 读取未提交内容(脏读，未提交的数据，可能回滚即数据最后不存在)
  • rc 读取已提交内容(不可重复读，数据不一致，有update操作)
  • rr 可重复读(可能出现幻读，即插入了一个数据行)
  • serializable 串行化

mysql默认rr: 在快照读读情况下，mysql通过mvcc来避免幻读; 在当前读情况下，mysql通过next-key来避免幻读。

3、MVCC原理

1、分库分表联表查询有哪些方式

其实可以考虑下架构设计，一切导致复杂度增高的设计是否是好设计是需要探讨的，设计好数据库，尽量避免这类查询情况的产生

1）分库时把需要关联的数据放到同一个库，尽量避免了夸库查询；
2）实在需要关联查询的，在一个库查出一部分数据，然后再查另一部分，自己在代码中组合；
3）用服务组合需要关联的数据，监视数据变化，将数据拿到进行组合放入 nosql 内存数据库，查询时直接从 nosql 内存数据库查询；
4）使用冗余字段，部分字段两个表都存，直接查询；
5）修改业务，不允许业务展示时关联不必要的数据，或者企图展示过多数据；

2、读扩散&写扩散

feeds流类型的业务中的问题

读扩散适用于在写多读少的场景，若读请求过多可能导致热点问题。

写扩散: 数据独立，中心点数据丢失对其他的无影响，但是浪费成本，数据实时性差

读扩散: 实时性高、降低存储成本，但是会有热点问题
