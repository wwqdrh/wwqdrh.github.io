什么是redis

一个内存型的key-value存储系统，支持多种数据结构包括字符串、哈希表、列表、集合和有序集合等，以及一些高级功能，如事务、发布/订阅、Lua脚本等。

支持集群、支持数据持久化

可以用于

  • 缓存： Redis可以作为缓存系统，提高Web应用程序的性能。
  • 会话管理： Redis可以作为会话管理系统，存储用户的登录状态和其他相关信息。
  • 消息队列： Redis可以作为消息队列系统，用于异步处理任务和消息。
  • 计数器： Redis可以作为计数器系统，用于统计网站的访问量和其他相关数据。
  • 实时排行榜： Redis可以用于实时排行榜系统，用于展示热门文章、商品等。
  • 分布式锁： Redis可以用于实现分布式锁，保证多个进程或线程的互斥访问。
  • 地理位置查询： Redis可以用于地理位置查询，用于搜索附近的商家、朋友等。

单进程多线程模型，所有的指令都串行的在同一个线程中执行达到原子性

redis的事务实现

Redis事务的实现主要通过MULTI、EXEC和WATCH、discard、unwatch，其中MULTI用于开启事务，EXEC用于提交事务、WATCH用于监视任意数量的key。

WATCH命令是一种乐观锁的实现方式，它在EXEC命令执行之前监视任意数量的键，并在EXEC执行期间检查监视的键是否至少有一个已经被修改。如果是，服务器拒绝执行事务，并向客户端返回代表事务执行失败的空回复。如果一个键都没有被修改，则成功执行事务。 WATCH命令监视的键会保存在数据库的watched_keys字典中，key为监视的键，value为监视该键的所有客户端

数据结构以及实现

Redis6支持多种数据结构，以下是其中一些常用的数据结构以及底层的实现方式：

字符串类型：字符串类型是Redis中最简单的数据结构，底层实现方式是一个字符数组。 哈希表类型：哈希表类型是一种键值对集合，底层实现方式是一个哈希表，其中每个键对应一个哈希桶，每个哈希桶中存储多个键值对。 列表类型：列表类型是一种可变长度的序列，底层实现方式是一个双向链表。 集合类型：集合类型是一种无序不重复的数据集合，底层实现方式是一个哈希表，其中每个键对应一个哈希桶，每个哈希桶中存储多个元素。 有序集合类型：有序集合类型是一种有序的数据集合，底层实现方式是一个跳表和一个哈希表，其中跳表用于存储元素和元素的分值，哈希表用于存储元素和元素的下标。 Bitmap类型：Bitmap类型是一种位图数据结构，底层实现方式是一个字符数组，每个字符存储8个位的值。 以上是一些常用的Redis数据结构及其底层实现方式，不同的数据结构在底层实现方式上也有所不同，因此在使用Redis时需要根据实际情况选择合适的数据结构。

string底层如何实现的

Redis6中字符串的底层结构是一个字节数组，其内部实现方式类似于C语言中的char[]数组。Redis6的字符串类型可以存储任意长度的二进制数据，包括文本、二进制数据、序列化对象等。字符串类型支持多种操作，如设置、获取、追加、自增等。

在Redis6中，字符串类型的底层结构是一个SDS（Simple Dynamic String）结构体，SDS结构体包含三个成员变量：

buf：一个指向字节数组的指针，用于存储字符串的值。 len：一个整数类型的变量，用于记录字符串的长度。 free：一个整数类型的变量，用于记录buf数组中未使用的字节数量，以便在需要扩展字符串时使用。 SDS结构体的设计使得Redis6的字符串类型具有以下优点：

动态扩展：由于SDS结构体的free变量可以记录buf数组中未使用的字节数量，因此在需要扩展字符串时可以快速地分配更多的内存空间，而无需频繁地进行内存分配和释放操作。 常数时间复杂度：由于SDS结构体中的len变量可以快速地获取字符串的长度，因此在获取字符串长度和进行字符串拼接等操作时，时间复杂度为常数级别。 二进制安全：由于SDS结构体可以存储任意长度的二进制数据，因此Redis6的字符串类型可以存储文本、二进制数据、序列化对象等多种数据类型。 总之，Redis6的字符串类型采用SDS结构体作为底层实现方式，具有动态扩展、常数时间复杂度和二进制安全等优点，是Redis6中最基本和最常用的数据结构之一。

列表底层是如何实现的

Redis6中列表的底层实现采用的是双向链表和压缩列表的混合实现方式。

双向链表是一种基本的数据结构，其每个节点包含一个指向前一个节点和后一个节点的指针，因此可以快速地在列表的头部和尾部进行插入和删除操作。Redis6中的双向链表是由listNode结构体实现的，listNode结构体包含三个成员变量：prev、next和value，分别用于指向前一个节点、后一个节点和节点的值。

压缩列表是一种特殊的数据结构，用于在存储空间和时间效率之间进行权衡。压缩列表将多个小节点的值存储在一起，以节省空间，同时也支持快速地访问列表中的任意元素。Redis6中的压缩列表由ziplist结构体实现，ziplist结构体包含多个字节数组，每个字节数组存储一个节点的值，以及一些元数据，如节点的类型、节点的长度等。

在Redis6中，当列表的元素数量较少时，采用双向链表的方式实现；当列表的元素数量较多时，采用压缩列表的方式实现。列表的底层实现方式由Redis自动根据列表的长度和元素类型等因素进行选择，以保证Redis的性能和空间效率。

Redis6中列表支持多种操作，如添加、删除、获取、修剪、阻塞等。列表的底层实现方式使得Redis6的列表类型具有以下优点：

灵活性：列表类型可以存储任意类型的元素，包括字符串、哈希表、集合等。 高效性：由于列表采用双向链表和压缩列表的混合实现方式，因此在插入、删除、获取元素等操作时具有较高的效率。 可靠性：由于Redis支持数据持久化，因此即使在Redis服务器宕机或重启时，列表中的数据也不会丢失。 总之，Redis6的列表类型采用双向链表和压缩列表的混合实现方式，具有灵活性、高效性和可靠性等优点，是Redis6中常用的数据结构之一。

有序集合底层是如何实现的

Redis6 中的有序集合（Sorted Set）是通过跳跃表（Skip List）和字典（Dictionary）两种数据结构来实现的。

在Redis中，跳跃表是一种基于有序链表的数据结构，它可以高效地实现插入、删除和查找操作。跳跃表的结构类似于多层链表，每层链表都是原链表的子集，并且每一层链表的节点数量会随着层数的增加而减少，从而达到了快速查找的效果。

在Redis中，跳跃表被用来实现有序集合中的成员和分值之间的映射关系。具体来说，有序集合中的每个成员和分值会被存储为一个跳跃表节点，节点中存储了成员和分值的信息，以及指向前驱和后继节点的指针。

另外，在Redis中，有序集合的成员和分值还会被存储到一个字典中。字典是一种高效的键值对存储结构，它可以快速地实现查找、插入和删除操作。在有序集合中，字典主要用来实现成员的唯一性约束和成员之间的快速查找。

因此，Redis6 中的有序集合底层是由跳跃表和字典两种数据结构共同实现的，通过跳跃表实现有序性和快速插入、删除和查找操作，通过字典实现成员的唯一性和快速查找操作。这种设计既兼顾了性能和空间效率，又保证了数据的一致性和正确性。

redis过期策略和内存淘汰策略

Redis的过期策略是指在Redis中设置一个key的过期时间，当这个key的过期时间到达时，Redis会自动将这个key从内存中删除。

Redis的内存淘汰策略是指当Redis的内存达到一定的限制时，Redis会自动淘汰一些不常用的key，以释放一部分内存。

redis的数据持久化方式有哪些

RDB 是快照方式持久化，通过将内存中的数据定期写入到硬盘上的文件中，实现数据的持久化。可以手动执行 SAVE 命令，也可以配置快照策略，比如 save 900 1 表示在 900 秒内发生至少一个 key 的变化，则执行快照。默认情况下，快照文件的名称为 dump.rdb，保存在 Redis 安装目录下。

AOF 是追加方式持久化，每一条写命令都会被以追加的方式写入到 AOF 文件中，以此记录 Redis 中的所有操作。可以通过配置 appendonly yes 开启 AOF 持久化。AOF 文件默认为 appendonly.aof，保存在 Redis 安装目录下。

可以同时开启 RDB 和 AOF，当 Redis 重启时，会优先加载 AOF 文件中的数据，如果 AOF 文件不存在，则加载 RDB 文件中的数据。

开启 RDB 持久化方式，在 Redis 配置文件 redis.conf 中修改以下属性：

save 900 1   // 在 900 秒内，至少有一个 key 发生改变，则保存快照
dir /var/lib/redis   // 快照文件保存路径，可以自定义

开启 AOF 持久化 Redis 配置文件 redis.conf 中修改以下属性：

appendonly yes   // 开启 AOF 持久化
appendfsync everysec  // 每隔一秒钟对 AOF 文件进行一次同步持久化
dir /var/lib/redis   // AOF 文件保存路径，可以自定义

修改完配置文件后，需要重启 Redis 服务使配置生效。可以通过命令 redis-cli bgsave 手动执行 RDB 持久化，通过命令 redis-cli bgrewriteaof 手动执行 AOF 持久化。

redis的setnx底层怎么实现的

redis主从同步怎么做的

当新加入一台从服务器时，数据同步是通过以下步骤完成的：

  • 新加入的从服务器与主服务器建立连接，发送 SYNC 命令请求同步数据。
  • 主节点收到 SYNC 命令请求后启动后台进程，将当前数据库的数据快照保存到 RDB 文件中，同时将执行期间收到的所有写命令缓存起来。
  • 主节点将 RDB 文件发送到从节点，从节点接收到 RDB 文件后使用自己的数据结构加载数据，加载成功后，从节点会向主节点发送一条 PSYNC 命令请求，PSYNC 命令携带两个参数：同步偏移量和 master id。
  • 主节点收到 PSYNC 命令请求后，如果从节点的同步偏移量大于主节点执行中的偏移量，那么主节点可以对从节点进行增量同步。否则，主节点会采用全量同步的方式来完成数据同步。
  • 如果主节点采用全量同步，从节点会再次向主节点 SYNC请求节点重新进行数据快照并发送给从节点。从节点在加载 RDB 文件时，记录下最后一条命令的执行偏移量作为自己的同步偏移量。
  • 如果主节点采用增量同步，主节点将执行期间收到的所有写命令发送到从节点，从节点按照顺序执行这些命令，完成数据同步。

集群模式下如何保证高可用

在Redis集群模式下，可以通过以下几种方法来保证高可用：

  • 数据复制：Redis集群的每个节点都会有多个复制节点，这些节点可以用来备份数据。当主节点宕机时，备份节点可以接管主节点的工作，保证数据不会丢失。
  • 自动故障转移：节点宕机时，Redis集群会自动将备份节点中的一个节点升级为主节点，这个过程称为自动故障转移。自动故障转移可以确保服务在主节点宕机时不会中断。
  • 客户端分片：Redis集群支持在客户端实现分片，使得客户端可以将数据分散地存储在不同的 Redis 节点上。这种方式可以提高系统的并发能力和负载均衡能力，避免单个节点过载。
  • 监控和报警：Redis集群需要能够及时监控节点是否正常工作，并能够通过报警系统及时通知管理员。监控和报警是保证系统高可用的重要手段之一。
  • 数据备份和恢复：为了防止数据丢失，Redis集群需要进行定期的数据备份，以便在出现故障时能够快速地进行数据恢复。

缓存失效的场景以及解决办法

缓存雪崩: 大量的key同一时间失效，导致请求全都直接达到数据库中，设置随机过期时间

缓存击穿: 与雪崩类似，不过这里是热点key失效，使用singleflight幂等请求，使得只有一个请求会进入数据库，或者业务允许的话，不为热点key设置过期时间

缓存穿透: 数据库和缓存中都不存在，可以缓存零值，或者使用布隆过滤器进行过滤

redis写入通信协议是什么

Redis协议RESP位于TCP层之上，即客户端和Redis实例保持双工的连接。

Redis客户端与服务器交互采用序列化协议（RESP）,请求以字符串数组的形式来表示要执行命令的参数。用命令特有（command-specific）数据类型作为回复。

通信协议的特点

  • 客户端和服务器通过 TCP 连接来进行数据交互， 服务器默认的端口号为 6379 。
  • 客户端和服务器发送的命令或数据一律以 \r\n （CRLF）结尾。
  • 在这个协议中， 所有发送至 Redis 服务器的参数都是二进制安全（binary safe）的。
  • 简单，高效，易读。

扩展类型说下

  • geo: 存储地理位置信息，能够计算两点距离，能够计算以某点为中心返回不超过给定距离的所有位置
  • bitmap: 位图，Bitmap底层使用String实现，value的值最大能存储512M字节，可以表示 512 * 1024 * 1024*8=4294967296个位，可以用于布隆过滤器，可以用于签到
  • hyperloglog: 用于基数统计的数据集合类型。它的最大优势就在于，当集合元素数量非常多时，它计算基数所需的空间总是固定的，而且还很小。
  • stream：实现消息队列，xadd、xrange、xdel、xget；xgroup、xreadgroup、xpending

Redis5.0的新数据结构, Stream 的消费模型借鉴了 Kafka 的消费分组的概念，它弥补了 Redis Pub/Sub 不能持 久化消息的缺陷。但是它又不同于 kafka，Kafka 的消息可以分 partition，而 Stream 不行。 如果非要分 parition 的话，得在客户端做，提供不同的 Stream 名称，对消息进行 hash 取 模来选择往哪个 Stream 里塞。

Stream有一个消息链表，将所有加入的消息都串起来，每 个消息都有一个唯一的 ID 和对应的内容。消息是持久化的，Redis 重启后，内容还在。

每个 Stream 都可以挂多个消费组，每个消费组会有个游标 last_delivered_id 在 Stream 数组之上往前移动，表示当前消费组已经消费到哪条消息了。每个消费组都有一个 Stream 内唯一的名称，消费组不会自动创建，它需要单独的指令 xgroup create 进行创建，需要指定 从 Stream 的某个消息 ID 开始消费，这个 ID 用来初始化 last_delivered_id 变量。

每个消费组 (Consumer Group) 的状态都是独立的，相互不受影响。也就是说同一份 Stream 内部的消息会被每个消费组都消费到。

同一个消费组 (Consumer Group) 可以挂接多个消费者 (Consumer)，这些消费者之间是 竞争关系，任意一个消费者读取了消息都会使游标 last_delivered_id 往前移动。每个消费者有 一个组内唯一名称。

消费者 (Consumer) 内部会有个状态变量 pending_ids，它记录了当前已经被客户端读取 的消息，但是还没有 ack。如果客户端没有 ack，这个变量里面的消息 ID 会越来越多，一 旦某个消息被 ack，它就开始减少。这个 pending_ids 变量在 Redis 官方被称之为 PEL，也 就是 Pending Entries List，这是一个很核心的数据结构，它用来确保客户端至少消费了消息一 次，而不会在网络传输的中途丢失了没处理。

限制用户评论过去一个小时内只能评论三次，用redis实现

  • 首先，我们需要使用 Redis 的有序集合（Sorted Set）来保存每个用户在过去一小时内的评论时间戳和次数。有序集合的键是用户ID，值是评论时间戳和次数的元组，其中时间戳是有序集合的分数，次数是元素的值。
ZADD comments:count:user_id timestamp1 count1 timestamp2 count2 ...

例如，以下命令将用户ID为user_id的用户的评论时间戳和次数插入有序集合中：

ZADD comments:count:user_id 1636000000 1 1635999600 2 1635999200 1

在上述示例中，我们将用户ID为user_id的用户的评论时间戳和次数插入有序集合中，其中1636000000、1635999600、1635999200是评论时间戳，1、2、1是评论次数。

  • 然后，每当用户尝试评论时，我们可以通过以下命令将当前时间戳和次数插入有序集合中，并使用 Redis 的ZREMRANGEBYSCORE命令将过期的评论时间戳和次数从有序集合中删除。
ZADD comments:count:user_id current_timestamp 1
ZREMRANGEBYSCORE comments:count:user_id 0 current_timestamp-3600

在上述示例中，我们使用ZADD命令将当前时间戳和次数插入有序集合中，并使用ZREMRANGEBYSCORE命令删除所有时间戳小于当前时间戳减去一小时的元素。

  • 最后，我们可以使用 Redis 的ZCARD命令获取有序集合中的元素数量，以检查用户在过去一小时内的评论次数是否超过了限制。
ZCARD comments:count:user_id
