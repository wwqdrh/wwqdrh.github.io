# 概览

dns服务查找相关

`/etc/hostname`: 表示在 LAN（局域网）内的唯一主机名

`/etc/hosts`: 为本地提供了 IP 地址与 hostname的对应

`/etc/resolv.conf`:  DNS 解析的配置文件，它指明了需要域名解析时的设置

```conf
nameserver 127.0.0.11 // 如果需要 DNS 解析的话，会去 127.0.0.11 询问。
options ndots:0

// 如果需要查找域名是相对域名，且该域名中包含的. 的数目大于或等于 option ndots:${n}命令指定的数，则查询的仅是该域名。否则会依次往传入的域名后追加 search 列表（search 列表在这里没有，详见search）中的后缀，直到解析出ip地址，或者解析完列表中所有后缀才会停止。这里设置成 0，意思就是对于传入的相对域名，只查询该域名
```

网络模型如下

- bridge: 默认使用的网络，可以允许 container 和外部网络以及其他 container 通讯， 不支持服务自动发现，没有容器名到ip的映射
- none: 不分配网络信息，无法进行通信
- host: 与宿主机同一个网络，也就是不会进行隔离
- 自定义bridge: `docker network create --driver bridge isolated_nw`, 通过docker内置的DNS服务实现容器名与ip的映射
- overlay：在不同宿主机的容器之间需要互相通信的情景
- macvlan


# overlay模式

`docker stack`, swarm模式下的默认网络模式

Swarm有Service的概念。一个Service是指使用相同镜像、同时运行的多个容器，多个容器同时一起对外提供服务，多个容器之间负载均衡。每个Service会有一个浮动IP（VIP），各个容器还有自己的物理IP。

创建基于Swarm的Overlay网络，将Service挂载到此网络上。然后Service中的各个容器便可以通过Service名称（同时也是一个DNS名称）和IP地址实现网络互通。

默认情况下，当创建了一个服务并连接到某个网络后，swarm会为该服务分配一个VIP。此VIP根据服务名映射到DNS。在网络上的容器共享该服务的DNS映射，所以网络上的任意容器可以通过服务名访问服务。
